# FDE 标准作业程序（SOP）
## Forward Deployed Engineer — 从进场到交付的完整操作手册

> **核心信条**：FDE 的本质是弥合"技术能力"与"业务价值"之间的鸿沟。代码只是手段，信任才是产品。

---

## Part 0：理解你的角色

### 0.1 FDE 是什么，不是什么

| 你**不是** | 你**是** |
|-----------|---------|
| 顾问——给PPT后离开 | 驻场工程师——亲手把代码跑起来 |
| 项目经理——管人管进度 | 端到端负责人——从需求到生产全程主导 |
| 传统研发——等需求文档 | 需求翻译官——把模糊业务痛点转化为技术问题 |
| 技术支持——救火修bug | 价值制造者——主动识别核心问题并解决 |

**一句话定位**：FDE 是一人扮演 工程师 + 产品经理 + 战略顾问 + 客户成功 四个角色的复合型工程师。

### 0.2 你的时间分配

根据行业实践，FDE 典型的时间分配：

| 工作类型 | 时间占比 |
|---------|---------|
| 写代码、集成、调试 | ~70% |
| 开会、与客户对齐、汇报 | ~20% |
| 文档、知识沉淀、反馈产品 | ~10% |

> ⚠️ 如果你的"会议"超过40%，你已经偏离FDE角色，在做咨询顾问的事了。

### 0.3 FDE 的成功指标

| 层级 | 指标 |
|------|------|
| **交付层** | MVD 在约定时间内上线，核心功能可被真实用户使用 |
| **信任层** | 客户愿意推动扩大合作范围 |
| **价值层** | 客户能量化说出"这个系统帮我节省了X小时/降低了X%错误率" |
| **产品层** | 将客户痛点反馈给产品团队，推动通用化 |

## Part 1：项目启动 SOP（进场前 & 第一周）

### 1.1 进场前准备清单

**技术环境准备（出发前必做）**

- [ ] 打包离线部署包：Python/Node运行时、所有依赖库（pip wheel/npm pack）
- [ ] 准备版本控制环境（Git）
- [ ] 确认访问客户环境所需的权限申请清单
- [ ] 了解客户的网络环境：内网/外网/VPN政策
- [ ] 准备轻量级Demo展示工具（Streamlit/Gradio或纯CLI均可）

**业务信息摸底（进场前1-2天）**

- [ ] 客户行业背景与主要业务模式
- [ ] 本次项目的直接对接人（技术侧+业务侧）
- [ ] 客户已有的IT系统栈（CRM/ERP/数据库类型）
- [ ] 项目的时间约束：**老板什么时候要看Demo？**
- [ ] 项目的直接触发原因（为什么现在启动？）

---

### 1.2 第一天操作清单（Day 1 Protocol）

**上午：发现（Discovery）**

| 时间 | 动作 |
|------|------|
| 9:00-10:30 | 与业务负责人访谈：了解他们的TOP3痛点 |
| 10:30-11:30 | 与技术负责人访谈：了解数据、系统、约束 |
| 11:30-12:00 | 整理访谈记录，形成一句话需求定义 |

**访谈时必问的五个问题**：
1. "如果这个系统能帮你做一件事，那件事是什么？"
2. "老板/领导最关心的业务结果是什么？"
3. "现在是怎么做这件事的？最大的麻烦在哪里？"
4. "你觉得什么时候能看到初步结果？"
5. "数据在哪里？我现在能看到吗？"

**下午：打通（Break Ground）**

| 时间 | 动作 |
|------|------|
| 13:30-14:30 | 在客户环境跑Hello World，验证环境可用 |
| 14:30-17:00 | 开始写核心路径代码（允许硬编码、允许丑） |
| 17:00-18:00 | 有一个"动"起来的东西给客户看，哪怕只跑通了一步 |

> ✅ **Day 1 的成功标准**：客户能亲眼看到"他的数据在你写的代码里动了"

### 1.3 MVD 边界划定（第1-2天）

**第一步：用三层优先级法梳理功能**

```
Must-have（必须交付）
├── 定义：没有它，客户看不到任何业务价值
└── 示例：数据接入 + 核心业务逻辑跑通 + 用户能完成一次完整操作

Should-have（应该交付）
├── 定义：有了体验更好，没有也不阻塞核心价值
└── 示例：美观界面、权限管理、操作日志

Nice-to-have（锦上添花）
├── 定义：有了更好，没有完全不影响
└── 示例：批量操作、Excel导出、邮件通知
```

**MVD = Must-have 全部 + 辅助验证的 Should-have 部分**

**第二步：与客户对齐 MVD 边界**

用这个句式确认边界：
> "老板要你在 [日期] 演示，你觉得演示中**必须出现**的功能是什么？少了哪个会让他觉得没做？"

**第三步：写下来，让客户签字或微信确认**

MVD 确认单模板：
```
MVD 范围确认（[项目名] [日期]）

必须交付（Must-have）：
1. [功能1]
2. [功能2]
3. [功能3]

不在本次范围（defer）：
- [功能X]
- [功能Y]

目标演示日期：[日期]
确认人：[客户联系人]
```

## Part 2：工程执行 SOP

### 2.1 核心工程节奏：三段式推进

```
阶段一：打通  →  阶段二：信任  →  阶段三：推进
（前三快）        （遇信慢）        （稳定加速）
```

---

#### 阶段一：打通（Phase 1 - Break Ground）

**目标**：最快速度跑通"用户输入 → 系统输出"这条最短路径

**这阶段的工程原则**：

| 原则 | 说明 |
|------|------|
| ✅ 允许硬编码 | 用 `# TODO: config` 标记，之后再提取 |
| ✅ 允许代码堆在一个文件 | 先跑通，再拆分 |
| ✅ 允许界面极简/无界面 | CLI输出比等前端框架快得多 |
| ✅ 允许跳过边缘情况 | 正常路径先跑通 |
| ❌ 不允许不写注释 | 你下周可能忘了这段代码在干嘛 |
| ❌ 不允许跳过核心路径验证 | 每跑通一步就在日志里留记录 |

**典型核心路径模式**：

```
智能问答：用户提问 → 检索知识库 → 返回答案
数据分析：数据导入 → 业务计算 → 结果输出
工作流自动化：触发事件 → 逻辑处理 → 动作执行
```

**阶段一完成标志**：
> 有一个Demo可以给客户看，哪怕慢、哪怕丑、哪怕只跑通了主流程

---

#### 阶段二：信任（Phase 2 - Build Trust）

**核心路径跑通后，必须在这三个维度上慢下来认真做：**

| 信任关键点 | 最低标准 | 为什么不能跳过 |
|-----------|---------|--------------|
| **数据安全** | 权限控制到位，敏感数据不明文存储 | 一次数据泄露 = 信任归零且不可修复 |
| **结果准确性** | 核心输出有验证机制，异常有提示 | 一次"瞎推荐/错误结果" = 用户永远不信这个系统 |
| **稳定性** | 核心路径不崩溃，允许慢但不允许500 | 客户演示时崩溃，比没做更糟 |

**安全检查清单**：
- [ ] 敏感字段（密码、Token、个人信息）是否加密存储
- [ ] API密钥是否移出代码（放到环境变量或配置文件）
- [ ] SQL查询是否参数化（防注入）
- [ ] 是否限制了文件上传类型和大小
- [ ] 关键操作是否有日志记录

**准确性验证清单**：
- [ ] 准备一组已知答案的测试用例
- [ ] 核心计算逻辑是否经过手工验证
- [ ] 异常输入是否有合理的错误提示
- [ ] 边界值（空输入、超大值）是否处理

---

#### 阶段三：推进（Phase 3 - Steady Advance）

完成前两个阶段后，进入MVD收尾阶段。

**每日节奏（Daily Rhythm）**：

| 时间段 | 内容 |
|--------|------|
| 每天早上 | 与客户对齐昨天的进展 + 今天的目标（15分钟站会） |
| 上午主力开发 | 聚焦当日最重要的一个功能 |
| 下午配对/测试 | 让客户侧的人一起看，尽早发现理解偏差 |
| 每天结束前 | 给客户发一条进展消息（哪怕只有一句话） |

**每日进展消息模板**：
```
[项目名] 今日进展 [日期]

✅ 完成：[今天完成了什么]
🔄 进行中：[明天继续推进什么]
⚠️ 需确认：[需要客户反馈/决策的事项]
```

---

### 2.2 工程取舍决策框架

**当你面临技术选择时，用这个框架决策：**

#### 框架1：优雅 vs 实用

```
问自己：这个代码跑在"核心路径"上吗？

核心路径上 → 可以不优雅，但必须正确、稳定
核心路径外 → 如果时间允许，写优雅一点
数据安全/权限 → 任何情况下不妥协
```

#### 框架2：完整 vs 可用

```
问自己：这个边缘情况在真实使用中会频繁发生吗？

频率高（主流程的一部分）→ 必须处理
频率低（偶发场景）→ 先记到tech_debt.md，反馈期再处理
完全不确定 → 先不做，等用户使用后看实际反馈
```

#### 框架3：自动化 vs 手动

```
问自己：这个操作在验证期会发生几次？

< 10次 → 手动做，别花时间自动化
每天都有 → 必须自动化
介于之间 → 手动跑1-2周，理解清楚了再自动化
```

> **原则**：先手动跑两周，再花三天写自动化，效果比一开始就自动化好得多——因为手动过程中你会发现真正的复杂性在哪里。

### 2.3 技术债务管理

**必须在项目目录维护 `tech_debt.md`**

```markdown
# Tech Debt Log

| # | 描述 | 优先级 | 计划处理时机 |
|---|------|--------|------------|
| 1 | 数据库连接字符串硬编码 | P1 | 上线前 |
| 2 | 用户列表手动维护 | P2 | 第一次反馈后 |
| 3 | 界面缺少加载状态提示 | P3 | Nice-to-have阶段 |
```

**可以欠的债**（记录即可）：
- 代码重复、配置硬编码、界面简陋、缺少部分日志

**绝不能欠的债**（必须立刻处理）：
- 敏感数据明文存储
- SQL注入漏洞
- 无法复现的部署流程
- 单点故障无备份

## Part 3：客户关系管理 SOP

### 3.1 利益相关方地图

进场第一天就要画出：

```
高层决策者（关心：ROI、业务结果、项目进度）
    ↓ 汇报
业务负责人（关心：功能是否满足需求、使用体验）
    ↓ 协作
IT/技术负责人（关心：系统安全、集成复杂度、运维成本）
    ↓ 配合
一线用户（关心：是否好用、能否减少工作量）
```

**对不同层级的沟通策略**：

| 对象 | 汇报频率 | 汇报内容 | 语言风格 |
|------|---------|---------|---------|
| 高层决策者 | 每周1次 | 里程碑进展 + 风险 + 价值量化 | 业务语言，数字说话 |
| 业务负责人 | 每天 | 今日进展 + 明日计划 + 待确认事项 | 功能语言，Demo优于文字 |
| IT/技术负责人 | 按需 | 技术方案 + 安全措施 + 部署方案 | 技术语言，重点讲安全合规 |
| 一线用户 | 每次迭代后 | 新功能演示 + 使用反馈收集 | 操作语言，步骤要具体 |

---

### 3.2 演示的正确姿势

**好演示的标准**：用户看完后**非常想立刻用起来**（而不是"嗯，不错"然后就没了）

**演示准备清单**：
- [ ] 使用客户**真实的数据**而非样例数据
- [ ] 先演示对客户最有价值的场景
- [ ] 准备一个"惊喜时刻"（让客户说"哇这也能做到？"）
- [ ] 预留时间让客户自己操作
- [ ] 记录所有现场反馈（录音或详细笔记）

**演示中要说的话**：
> "这是你们的[真实数据]，系统帮你做了[具体操作]，你现在只需要[一个动作]就能得到[结果]。这和你们现在的做法相比，节省了[时间/步骤]。"

**演示后立刻做的事**：
1. 记录所有反馈（正面和负面都要记）
2. 当场确认：哪些反馈会纳入下一个版本，哪些defer
3. 约定下一次演示时间

---

### 3.3 需求变更管理

FDE 必然会遇到需求变更，处理原则：

| 变更类型 | 处理方式 |
|---------|---------|
| 小修小改（1-2小时内能做完）| 当天处理，更新进展消息 |
| 中等变更（半天到一天） | 评估对MVD时间线的影响，告知客户 |
| 大变更（影响核心架构） | 必须开专题讨论，重新对齐MVD范围 |
| 超出本次MVD范围 | 记录到"第二阶段需求列表"，不放入当前迭代 |

**说"不"的正确方式**：
> "这个功能我理解，但如果加进来会推迟[日期]的演示。我建议先把它放到第二阶段，确保[核心功能]按时交付，你觉得可以吗？"

## Part 4：部署上线 SOP

### 4.1 部署前检查清单（Production Readiness）

**代码层面**：
- [ ] 所有配置从硬编码提取到配置文件或环境变量
- [ ] 去掉所有调试日志和测试数据
- [ ] 核心路径有适当的错误处理和错误提示
- [ ] 数据库有备份方案
- [ ] 部署步骤已文档化（让一个不了解项目的人能按文档完成部署）

**安全层面**：
- [ ] 敏感信息（密码/Token/密钥）不在代码仓库里
- [ ] API接口有基本的认证保护
- [ ] 数据库连接使用最小权限原则
- [ ] 用户数据访问有审计日志

**运维层面**：
- [ ] 服务有健康检查接口
- [ ] 有基本的错误告警（即使只是发条微信也行）
- [ ] 有服务重启方案（systemd/supervisor/docker restart）
- [ ] 有回滚方案（怎么快速回到上一个稳定版本）

---

### 4.2 部署执行步骤

```
Step 1：在测试环境完整走一遍部署流程
    ↓ 发现并修复部署文档的问题
Step 2：通知所有相关方（客户IT + 业务负责人 + 一线用户）
    ↓ 确认维护窗口时间
Step 3：备份现有数据（如果是升级部署）
    ↓ 确认备份可用
Step 4：执行部署
    ↓ 每一步记录日志
Step 5：冒烟测试（核心功能验证）
    ↓ 至少走一遍完整的核心流程
Step 6：通知客户上线，陪同第一批用户使用
    ↓ 实时响应任何问题
Step 7：上线后48小时内保持高响应
    ↓ 主动收集第一批用户反馈
```

## Part 5：知识沉淀与反馈产品 SOP

### 5.1 每个项目结束必须输出

**客户侧文档**：
- [ ] 操作手册（一线用户能自己看懂的版本）
- [ ] 运维手册（IT管理员能自己处理常见问题）
- [ ] 常见问题FAQ

**内部文档**（反馈给产品团队）：
- [ ] 本项目的客户痛点摘要（哪些痛点是普遍的？）
- [ ] 在客户现场写的定制代码（哪些可以产品化？）
- [ ] 遇到的技术障碍（哪些是平台缺失的能力？）
- [ ] 本项目的价值量化（客户能说出多少ROI？）

### 5.2 反馈产品团队的正确方式

**不要说**：
> "客户想要一个[具体功能]"

**要说**：
> "我们观察到三个客户都有[这类场景]，我在每个项目里都写了类似的定制代码。核心需求是[抽象描述]，如果产品层面能支持，预计可以节省每个FDE项目X天的工作量。"

> **关键**：FDE反馈的是**通用化的场景**，不是客户的具体功能请求。这是FDE与普通定制开发的本质区别。

## Part 6：典型反模式（踩坑清单）

| 反模式 | 后果 | 正确做法 |
|-------|------|---------|
| 进场第一天花时间做完整架构设计 | 两周后发现架构和需求不匹配 | 第一天就写代码，第一个Demo出来后再讨论架构 |
| 等所有需求明确再开始写代码 | 需求永远不会"完全明确" | 基于当前理解开始，边做边对齐 |
| 先建完整的用户权限系统 | 两周过去，核心业务逻辑还没跑通 | 先跑通核心逻辑，权限后期加 |
| 追求生产级代码质量从第一天开始 | 速度慢，客户等不及 | 核心路径快速打通，加固期再重构 |
| 只和技术负责人沟通 | 技术上实现了，业务层面没价值 | 同步维护业务侧和技术侧两条沟通线 |
| 做完才给客户看 | 做偏了，全部推倒重来 | 每天给客户看进展，哪怕只是一个命令行输出 |
| 覆盖所有边缘情况再上线 | 永远上不了线 | 覆盖80%主流场景，剩余靠运营期反馈 |
| 把客户说的功能全部做进去 | 偏离核心价值，什么都有但什么都不好用 | 区分"客户说的"和"客户需要的"，聚焦核心 |

## 附录A：FDE 核心技能地图

```
技术能力（基础）
├── 编程：Python / TypeScript（必须），Java / Go（加分）
├── 数据：SQL高级查询、ETL管道、数据建模
├── API：REST / GraphQL、API集成与调试
├── 云与部署：Docker、K8s基础、AWS/Azure/GCP之一
└── AI/ML（现代FDE加分项）：
    ├── LLM调用与提示工程
    ├── RAG管道搭建（向量数据库、检索逻辑）
    ├── AI工作流集成（LangChain/LangGraph等）
    └── AI系统评估与可观测性

业务能力（差异化）
├── 需求挖掘：把模糊业务痛点转化为技术问题定义
├── 价值量化：用业务语言描述技术价值
├── 利益相关方管理：在不同层级切换沟通风格
```
