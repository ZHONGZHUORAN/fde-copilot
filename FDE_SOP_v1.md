# FDE 工程阶段 SOP
## 最小可行交付（MVD）标准作业程序

> 核心信条：FDE 的工程过程，就是**制造确定性**。代码只是手段，不是目的。

---

## 一、什么是 FDE？理解角色定位

FDE（Field Deployment Engineer / Forward Deployed Engineer）是直接驻扎在客户现场、负责把解决方案从原型推进到生产运行的工程师角色。

FDE 与传统工程师最大的区别：

| 维度 | 传统工程师 | FDE |
|------|-----------|-----|
| 关注点 | 代码优雅度、架构完整性 | 系统能不能在约定时间跑起来 |
| 工作节奏 | 需求分析 → 设计 → 开发 → 测试 | 第一天就写代码，当天出 Demo |
| 客户关系 | 通过 PM 间接沟通 | 每日与客户直接对齐 |
| 衡量标准 | 代码质量、覆盖率 | 确定性积累、客户信任建立 |

---

## 二、核心概念：MVD vs MVP

### MVD（Minimum Viable Delivery，最小可行交付）

**适用场景**：已知客户，需要在最短时间内让客户用上

**三个关键限定词**：

1. **给定的时间约束下** — MVD 的边界由"客户能等多久"决定，不是"理想的功能集合"
2. **能产生可验证业务价值** — 必须部署在客户环境，至少一个真实用户完成一次有意义的操作
3. **最小** — 砍掉一切不能产生直接价值的"锦上添花"

> "最小"是一种纪律，不是偷懒。

### MVD vs MVP 区别

| | MVP | MVD |
|-|-----|-----|
| 目标 | 验证假设 | 兑现承诺 |
| 面向 | 未知市场 | 已知客户 |
| 核心问题 | 产品-市场匹配吗？ | 系统能在约定时间跑起来吗？ |

## 三、MVD 边界划定方法：三层优先级法

```
Must-have（必须交付）
  ↓ 没有它，客户看不到任何业务价值
  示例：数据对接完成 + 核心逻辑跑通 + 客户能完成一次完整操作

Should-have（应该交付）
  ↓ 有了体验提升，没有也不影响核心价值验证
  示例：美观界面、用户权限管理、操作日志

Nice-to-have（锦上添花）
  ↓ 有了更好，没有无妨
  示例：批量操作、PDF 导出、邮件自动推送
```

**MVD = Must-have 全部 + Should-have 中辅助验证核心价值的部分**

### 与客户对齐 MVD 的有效对话

> "老板要你下周三演示，你觉得什么东西必须在演示中看到？"

客户的回答就是 MVD 的底线。

## 四、工程节奏 SOP：前三快 → 遇信慢 → 稳定加速

### 口诀
**"前三快，遇信慢，稳定加速稳交付。"**

---

### Step 1：前三快 — 核心路径用最快速度打通

**目标**：跑通"从用户输入到系统输出"的最短路径

**核心路径示例**：
- 智能客服项目：`用户输入问题 → 检索知识库 → 返回答案`
- 数据分析平台：`数据导入 → 核心计算逻辑 → 结果展示`

**这阶段允许**：
- [x] 硬编码（但加 `# TODO: move to config`）
- [x] 忽略边缘情况
- [x] 所有逻辑写在一个文件里
- [x] 界面极简甚至丑

**这阶段不做**：
- [ ] 用户管理界面（手动在后台添加用户）
- [ ] 邮件通知（微信群里通知）
- [ ] 数据可视化大屏（先导出 CSV 看数字）

**交付物**：一个可以给客户看的、跑通核心路径的 Demo

---

### Step 2：遇信慢 — 信任关键点必须认真对待

核心路径跑通后，必须慢下来处理这三类"信任关键点"：

| 关键点 | 为什么重要 | 最低标准 |
|-------|-----------|---------|
| **数据安全** | 一次数据泄露 = 信任归零 | 权限控制到位，敏感数据加密存储 |
| **结果准确性** | 一次"瞎推荐" = 信任归零 | 核心逻辑有验证机制，异常有提示 |
| **稳定性边界** | 核心功能不能崩溃 | 可以慢，但不能报错 |

**原则**：FDE 可以接受"系统不好用"，但不能接受"系统不可信"。

---

### Step 3：稳定加速 — 推进到 MVD

信任关键点处理完，保持稳定节奏推进剩余功能，直到 MVD 完成。

此时你有底牌：客户已经看到"东西在动"，节奏可以适度放慢，做得更稳。

---

## 五、工程取舍的三个智慧

### 5.1 优雅 vs 实用

**判断原则**：
- 核心路径上：可以接受不够优雅
- 数据安全和稳定性上：**绝不妥协**

### 5.2 完整 vs 可用

- "完整"：覆盖所有你能想到的边缘情况
- "可用"：覆盖 80% 的真实场景

**FDE 追求"可用"** — 因为反馈阶段会告诉你剩下 20% 里哪些真正重要。你现在猜测的边缘情况，可能 80% 都不会发生。

### 5.3 自动化 vs 手动

**判断标准**：
- 这个操作在验证期间只会发生 **< 10 次** → 手动做
- 这个操作**每天都会发生** → 必须自动化

> 不要一上来就做全自动流水线。FDE 手动帮客户处理两周的数据，效果比花三周开发自动化模块更好——因为两周后你对这个操作有了深刻理解，到时候做自动化会做得更好。

## 六、客户环境部署三个铁律

### 铁律一：第一天就验证环境

**不要**：第一天就部署完整系统  
**要**：在客户环境中跑通一个"Hello World"

一个最简单的脚本在客户服务器上成功运行，就证明了环境是通的，剩下的只是增量叠加。

### 铁律二：永远准备离线部署包

客户环境很可能是**内网**，无法访问外网。提前把所有依赖包打包好。

**检查清单**：
- [ ] 运行时环境（Python/Node 等）离线安装包
- [ ] 所有 pip/npm 依赖的离线包
- [ ] 数据库/中间件离线安装包
- [ ] 部署脚本和说明文档

### 铁律三：脏数据先入仓

**不要**：在数据对接阶段就把所有数据清洗干净  
**要**：先把数据原样拉到你的系统里，然后在你的系统里做清洗和转换

## 七、技术债务管理 SOP

### 可以欠的债

| 债务类型 | 处理方式 |
|---------|---------|
| 代码重复（同样逻辑复制三份） | 欠着，反馈阶段重构 |
| 硬编码配置 | 欠着，但加 `# TODO: move to config` 注释 |
| 界面简陋 | 完全可以欠，反馈阶段根据用户反馈美化 |

### 绝不能欠的债

| 债务类型 | 原因 |
|---------|------|
| 敏感数据明文存储 | 安全底线，零容忍 |
| SQL 注入漏洞 | 必须参数化查询 |
| 无备份的单点故障 | 必须设计备份方案 |
| 无法复现的部署流程 | 必须写清楚部署文档 |

### 技术债管理工具

在项目目录下维护 `tech_debt.md`：

```markdown
# tech_debt.md

| 描述 | 优先级 | 计划偿还时机 |
|------|--------|------------|
| 用户配置硬编码在代码里 | P2 | 反馈阶段第一周 |
| 数据清洗逻辑未模块化 | P2 | 生产上线前 |
```

## 八、Palantir 5 天极限工作法参考

这是 FDE 工程节奏的最佳实践参考：

| 时间 | 动作 | 产出 |
|------|------|------|
| Day 1 早 | 获取客户一句话需求 | 需求理解 |
| Day 1 午 | 开始写代码 | 第一行代码 |
| Day 1 晚 | 拿出第一个能跑的 Demo | 可演示的核心路径（丑没关系）|
| Day 2-4 每天 | 客户试用 → 给反馈 → FDE 迭代 | 每日版本 |
| Day 5 下午 | 向客户管理者演示 | 系统"能用了" |

**为什么这样工作**：
1. **压缩反馈环**：从几周压缩到几小时
2. **适应变化**：Day 2 发现"其实我想要 XX"——立刻改，成本极低
3. **建立信任**：客户看到"这个人在动"，不是"这个人每天都在开会"

## 九、"确定性"积累清单

FDE 的核心价值在于不断积累"确定性"，每一个里程碑都在消除客户焦虑：

| 里程碑 | 确定性增量 | 客户心理变化 |
|-------|-----------|------------|
| 数据管道跑通 | +1 | "至少数据能用了" |
| 核心功能 Demo 跑起来 | +3 | "逻辑是通的" |
| 客户测试环境部署成功 | +5 | "不是只在你电脑上能跑" |
| 第一个真实用户完成操作 | +10 | "这东西真的有用" |

## 十、常见反模式（避免踩坑）

| 反模式 | 正确做法 |
|-------|---------|
| 两周需求分析后才写第一行代码 | 第一天就写代码 |
| 先做完整的用户权限系统再展示 | 先展示核心功能，权限后期加 |
| 等数据清洗完再做对接 | 脏数据先入仓，系统里清洗 |
| 花三周做自动化流水线 | 先手动跑两周，积累理解再自动化 |
| 追求完美界面再给客户看 | 界面丑没关系，核心逻辑通了就展示 |
| 把所有边缘情况都覆盖再上线 | 覆盖 80% 真实场景，剩余靠反馈 |
